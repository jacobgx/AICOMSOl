# AICOMSOl 项目架构

## 设计目标

让 AI 编程助手能有效地进行 COMSOL 多物理场仿真建模。核心思路：**结构化知识库 + 铁律约束 = AI 可靠建模**。

## 三层架构

```
┌──────────────────────────────────────────────┐
│           Layer 3: 规范与控制层               │
│  CLAUDE.md (6条铁律 + 双轨策略 + 编译命令)     │
│  对 AI 的行为硬约束：检索先行、分步构建、      │
│  三级排查、API 逐行注释                       │
└──────────────────────────────────────────────┘
                      ↑ 查询
┌──────────────────────────────────────────────┐
│           Layer 2: 知识库                     │
│  database/                                   │
│  API参考 + 经验卡片 + 模型卡片 + 官方源码      │
│  建模前必须检索，严禁凭记忆写 API              │
└──────────────────────────────────────────────┘
                      ↑ 构建
┌──────────────────────────────────────────────┐
│         Layer 1: 输出与运行时                  │
│  result/                                     │
│  Java源码 + .mph模型 + Python脚本              │
│  双轨运行：javac → comsolbatch / MPh Python    │
└──────────────────────────────────────────────┘
```

## Layer 1 — 输出与运行时

`result/` 目录按项目组织，每个子目录是一个独立的多物理场建模项目：

```
result/
├── <ProjectName>/
│   ├── <Model>.java          # Java 模型源码
│   ├── <Model>.mph           # COMSOL 模型文件
│   ├── *.py                  # MPh 调试/后处理脚本
│   └── *.png                 # 结果可视化
```

运行时通过两条轨道执行建模：

### 轨道一：Java batch（新建模型）

```
手写 .java ──javac──→ .class ──comsolbatch──→ .mph + 结果
                 ↑ 编译时检查              ↑ 无GUI批量执行
```

适用场景：
- 从零创建新模型（几何 + 物理场 + 网格 + 求解）
- 需要完整的模型构建控制
- 拓扑优化等复杂工作流

### 轨道二：MPh（调试已有模型）

```
Python 加载 .mph ──→ 检查/改参 ──→ model.solve() ──→ model.evaluate()
       秒级反馈        不改代码        迭代收敛          直接拿数据
```

适用场景：
- 已有 .mph 改参求解
- 收敛调试（改参→求解→评估，5秒/轮）
- 参数扫描和灵敏度分析

## Layer 2 — 知识库

知识库是 AI 做出正确决策的信息基础，包含 4 大知识源：

| 知识源 | 位置 | 内容 | 规模 |
|--------|------|------|------|
| API 参考 | `database/api_reference/` | Java API 方法签名 + MPh 对照 | 4 文件, 1470 条 |
| 调试经验 | `database/experience/` | 错误诊断 + 收敛改善 + 领域知识 | 3 文件 |
| 官方模型卡 | `database/model_cards.json` | 结构化模型描述 | 1810 张 |
| 官方示例 | `database/applications/` | Java 源码，按模块组织 | 49 模块 |

### 知识检索流程

```
用户查询 ("层流 2D 入口边界")
    │
    ├─→ grep model_cards.json（模型名/描述/物理场）
    ├─→ grep api_reference/（API 语法确认）
    ├─→ grep experience/（常见错误排查）
    └─→ ls applications/<Module>/（示例代码参考）
```

## Layer 3 — 规范与控制层

CLAUDE.md 定义了 6 条铁律，每条铁律都有明确的触发条件和违规后果：

| # | 铁律 | 触发条件 | 行为约束 |
|---|------|----------|----------|
| 一 | 查 API 参考，每行加注释 | 写任何 API 调用前 | 先查参考再写代码 |
| 二 | 两轮修复失败检索网络 | 连续 2 次修复失败 | 停止改代码，搜索外部资料 |
| 三 | 修改模型只改参数 | 修改已有 .java 文件 | 完整复制，只改参数值 |
| 四 | 复杂模型分步构建 | 多物理场耦合任务 | 几何确认→单场→逐步耦合 |
| 五 | 按三级排查收敛 | 求解失败/不收敛 | 物理场一致性→Log→收敛改善 |
| 六 | batch 模式 API 限制 | Java batch 模式 | 使用替代 API |

## 双轨策略决策树

```
接到建模任务
    │
    ├─ 已有 .mph 文件？
    │   ├─ 是 → 只需改参？    → comsolbatch -pname (最快)
    │   │     → 需要调试？    → MPh (5s/轮迭代)
    │   │     → 需要检查配置？ → MPh Python 探针
    │   │
    │   └─ 否 → 新建模型     → Java batch
    │             ├─ 从零建几何  → Java batch (MPh 几何 API 不足)
    │             ├─ 拓扑优化    → Java batch (MPh batch 矩阵分解失败)
    │             └─ 其他        → 按需选择
    │
    └─ 查询知识库？
        └─ 总是 → grep database/
```
